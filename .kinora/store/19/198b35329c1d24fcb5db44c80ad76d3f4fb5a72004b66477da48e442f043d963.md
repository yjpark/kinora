# Link format: strong in-content links + weak in-metadata links

Settles RFC-0003 Q6 for the kinora bootstrap: how kinos link to each other.

## The decision
- **Chosen: two link venues by intent.** Strong (in-content) links are part of the content hash and parsed by the kind-specific renderer — kinograph `entries[]`, and markdown via a `kino://<id>/` URL scheme using native reference-link syntax. Weak (in-metadata) links live in `metadata.links: [{type, id, name?, pin?, note?}]` — queryable and backlinkable.
- **Chosen: `kind` = content type.** Bare kinds are Kinora core (`markdown`, `text`, `binary`, `kinograph`); extensions are namespaced (`kudo::diagram`, `user::sketch`). One field collapses kind + subtype cleanly.
- **Chosen: standard markdown reference links with `kino://<id>/` URLs** — no Kinora-specific markdown syntax invented, preserving markdown compatibility.
- **Rejected: name-based links / a bespoke markdown link syntax.** Names aren't authoritative; a custom syntax breaks markdown tooling. The strong/weak split matches natural authoring intent.
- **Not a dead end:** backlinks are derived from a ledger scan (metadata links) + content parse (in-content links); a cache-layer index is a post-bootstrap addition.

## What this built
- MVP kinds with parsers/renderers: `markdown` and `kinograph`. `text`/`binary` are storeable but opaque in render. `kino://<id>/` URL rewriting realized in [[kinora-9nom:task]]; kinograph entries in [[kinora-zboo:spec]].
