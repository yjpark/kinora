# Task: `root` reserved kind + flat root entry schema

**Goal.** Add the `root` reserved kind and a `RootKinograph` type whose flat entries (one per owned kino, with inline metadata) carry `id`, `version`, `kind`, `metadata`, optional `note`/`pin`.

**Realizes:**
- [[kinora-xi21:spec-2]] — Canonical history is a chain of root kinograph versions, with `root` as a reserved kind.

**Done when:** `root` is reserved with `ext_for_kind("root") == Some("styx")`; root entries parse and validate (valid hashes, valid kind, valid metadata keys, duplicate-id rejected); and parse → `to_styx()` → parse round-trips to canonical form (entries sorted by id, metadata keys sorted).

## Outcome
Resolved. Shipped a separate `crates/kinora/src/root.rs` module (`RootEntry`/`RootKinograph`) rather than extending the composition `Entry` — a deliberate spec departure, because root `pin` is a boolean GC marker while composition `pin` is a version hash, and callers always know which kind they parse. Auto-detect-by-`kind` was rejected for explicit module selection. Incidental finding: facet-styx handles `bool` and `BTreeMap<String,String>` nested in a `Vec<struct>` cleanly. Commits `ae86151`, `a491b45`, `a0abeea`.
