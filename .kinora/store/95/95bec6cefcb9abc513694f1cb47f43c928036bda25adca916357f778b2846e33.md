# Multiple named roots, each with its own GC policy; inbox is the default

Settles how a repo partitions its kinos for retention. Builds on the chained-root-kinograph history model and the main-only compaction discipline.

## The decision
- **Chosen: repos declare any number of named roots in `config.styx`, each with its own GC/prune policy** (`never`, `MaxAge` like `30d`/`12h`, `keep-last-N`), over a single global retention policy. Topical partitioning lets aggressive policies coexist with permanent ones.
- **Chosen: `inbox` is the auto-provisioned default root** for unassigned kinos, with an aggressive default policy (`30d`) to nudge triage discipline. Auto-provisioned if not declared.
- **Chosen: `kinora compact` always compacts every declared root; no `--root` flag (D5).** Per-root errors do not block clean roots — each root advances independently; the process exits non-zero iff any root errored, but clean roots still advance to disk. Library split into `compact_root` (per-root, testable) and `compact_all` (batch, CLI-facing).
- **Rejected: a single flat root / global policy.** It cannot express per-topic retention.
- **Rejected: auto-creating roots referenced by an assign (D4).** Declaring a root is an intentional config change, not a side effect of compact; an assign to an undeclared root fails loud with `UnknownRoot`.
- **Not a dead end:** config is not hot-reloaded — a config change takes effect on the next compact.
