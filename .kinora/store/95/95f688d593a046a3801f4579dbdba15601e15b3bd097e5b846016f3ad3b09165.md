# Kinora bootstrap: a minimal file-based knowledge system via CLI

The kinora bootstrap settles what the first usable version of Kinora is: a minimal, file-based system driven by a CLI, built per RFC-0003. This spec is the umbrella decision that frames the bootstrap cluster and its deliberately transitional storage model.

## The decision
- **Chosen: ship a minimal file-based Kinora usable end-to-end via CLI**, in dependency order: resolve RFC-0003 open questions → data layer (content store + ledger) → `store` → `resolve` → kinograph composition → `render` (mdbook) → dogfood by migrating kudo RFCs.
- **Chosen: the bootstrap's ledger-per-kino / one-lineage-per-repo layout is explicitly transitional.** It is the simplest thing that works for the bootstrap; the canonical history model supersedes it.
- **Rejected: building the canonical hot-ledger model up front.** Too much architecture before the store/resolve/render loop is proven by dogfooding.
- **Not a dead end:** the post-bootstrap architecture (one-file-per-event hot ledger, chained root-kinograph snapshots, a `root` kind, compaction on main, multiple named roots with `inbox` as default) is captured in kino://b0e9efc88034231698e376e4ffd496b437792b47b627c97cc498b99a589ba708. Once the bootstrap CLI is solid, the system evolves along that path.
