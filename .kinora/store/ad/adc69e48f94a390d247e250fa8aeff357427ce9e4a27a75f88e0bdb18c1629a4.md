# Compaction is a manual, deterministic, main-only operation

Settles the commit discipline: when and how hot events are promoted into new root kinograph versions.

## The decision
- **Chosen: compaction runs only on the configured main branch, via an explicit `kinora compact` command — no git hooks.** Other branches accumulate hot events as speculative proposals; render shows branch-only events as temp versions vs main's sealed versions. Explicit invocation keeps bootstrap simple and avoids pre-commit/post-commit/rebase edge cases. Doubled commits (source-intent + compaction) are acceptable.
- **Rejected: git post-commit/post-merge hooks for phase 2.** Hook wiring is deferred to a later phase only if manual invocation proves insufficient in practice.
- **Chosen: compaction is a pure function of `(hot_events, prior_root_version, root_config)` and must be byte-deterministic.** Entries sorted by id; canonical styx serialization (sorted keys, stable whitespace); on a unioned merge, the new version's `parents[]` lists both prior root versions in canonical hash order. Two developers compacting the same hot-event set independently must produce byte-identical root blobs, else branches diverge pointlessly on merge.
- **Chosen: genesis root is lazy and has no sentinel.** Identical shape to any other version, just empty `parents[]`. Created on the first `kinora compact` that has content to promote — no eager empty-kinograph blob at `kinora init`. `inbox` follows the same rule once multi-root lands.
- **Not a dead end:** hooks can be added later; the manual path is the conservative default.

## What this built
- `kinora::compact` library (`compact()`, `build_root()`, `read_root_pointer()`, `CompactResult`, `CompactError`), the `kinora compact` CLI with git-resolved author and RFC3339 timestamp, and `roots/<name>` pointer files (tmp+rename). Verified by genesis/subsequent/idempotence cases, a cross-dev byte-identical-blob determinism test, entry-sort and pointer-format tests, and an integration test (store 3 kinos → compact → bump → compact). Root-kind events are excluded from the root's own entries; forks error as `MultipleHeads`. See kino://44115de1bbeab63694497088a9128408a064622dece971abc001ccf08f24ced0.

## Consequences / follow-ups
- Phase 3 split `compact` into `compact_root` (per-root) and `compact_all` (batch) and retired the `--root` flag; see kino://50d1260aba15f848bb3656667a42f08ebcff7e3bcf3fb1d815f24ae6399ddd24.
- The single-parent happy path shipped; multi-parent post-merge root linking is a future path.
