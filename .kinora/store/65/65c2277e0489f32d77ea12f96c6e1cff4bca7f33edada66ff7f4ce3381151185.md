# Reverse extract: a read-only edit becomes a new spec kino version

A proposal (not yet committed) for the reverse stencil path: when an agent edits
inside a read-only region intentionally, treat it as an API change rather than a
process violation.

## The decision (proposed)
- **Proposed: `stencil extract` turns an intentional read-only-region edit into
  a new `kudo::api-spec` kino version + an api-kinograph update**, instead of the
  default hash-mismatch "hand-edited region" warning. This smooths the
  API-change UX — the source becomes a valid place to author a change, which
  stencil folds back into the kino model.
- **Rejected (status quo): treating every read-only edit as drift to overwrite.**
  Loses the edit and forces the agent to round-trip through `kinora store`
  manually; the reverse path keeps the edit and versions it.
- **Open / not a dead end:** how an edit is recognized as *intentional* vs
  accidental drift (explicit `stencil extract` command vs heuristic) is
  unresolved; this is filed as a forward direction, low priority.

## What this would build
- A reverse engine path: diff the read-only region against the rendered spec,
  synthesize a new spec kino, and update the bound api-kinograph entry.
