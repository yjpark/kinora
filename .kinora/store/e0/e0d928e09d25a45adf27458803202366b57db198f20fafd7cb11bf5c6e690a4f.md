# Task lifecycle semantics for kinora-as-tracker

Using kinora as a task tracker needs status/done/epic semantics; kinora has none
(currently proxied with prose in a plan kino plus the root GC policy). This is a
proposed contract for adding a task lifecycle, filed from dogfooding. Not yet
committed — it must be designed together with the GC-waterline and typed-link
proposals it overlaps.

## The decision
- **Proposed: design task-lifecycle alongside kino://4b0c148ab55499d1657eacc03f793e989c59db4584257525207d86feb165f8fd (GC waterline)
  and kino://e298936a4101424c40e9fb074339e73b7c44c1b8a1457b8a32cd5460589bf0b7 (typed links) as one "agentic tracking" design pass under
  the architecture epic kino://7d1a10e95dcee077fe2cc80f492c9f6ff299b490342800ab65f80f811a35e894** — rather than bolting status/done/epic onto
  the data model in isolation. These three are entangled and a piecemeal addition
  would calcify the wrong shape.
- **Open — where status lives:** metadata (`status=todo|doing|done`, reusing the
  reserved-key mechanism) vs. a dedicated event kind vs. a root-per-status. To be
  settled by the joint design.
- **Open — how "done" tasks sink:** this is exactly the GC-waterline question
  (kino://4b0c148ab55499d1657eacc03f793e989c59db4584257525207d86feb165f8fd) — done tasks route to a short-term root (maxage/keep-last-N),
  specs stay durable (never).
- **Open — epic/parent relationships:** likely reuse the typed-link primitive
  (kino://e298936a4101424c40e9fb074339e73b7c44c1b8a1457b8a32cd5460589bf0b7); blocked on that contract.
- **Not a dead end:** nothing is committed; the proposal stays `draft` until the
  joint design pass resolves the open questions.
