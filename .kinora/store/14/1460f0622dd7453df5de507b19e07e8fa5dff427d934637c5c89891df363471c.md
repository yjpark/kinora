# Root ownership is exclusive and structural; kinos move via assign events

Settles where a kino's metadata lives, how membership is expressed, and how a kino moves between roots. Depends on the named-roots model.

## The decision
- **Chosen: a kino's metadata home is its leaf position in its owning root's tree — ownership is structural, not declared.** Entries inside a root tree inline full metadata (`name`, `title`, …); kinographs outside any root (user composition kinographs) carry pure `{id, version}` pointers. Readers resolve "current metadata for kino X" by finding X's owning root and reading the inline metadata at its leaf.
- **Chosen: kinos never declare root membership.** No `root=X` on a kino — identity is the sole stable property. The relationship flows owner→owned: roots claim kinos, not the reverse. Ownership is exclusive (a kino lives in exactly one root) but composition is preserved (a kino may appear in many kinographs).
- **Chosen: moving between roots is an explicit `assign` event consumed by compaction.** An assign records "kino X should live in root Y from now on," carries an optional `supersedes` list, and is resolved at compact time. Birth into a non-default root writes a store+assign atomic pair (D3); plain `store` routes to inbox implicitly.
- **Chosen: conflicts fail loud (D2/D4).** ≥2 live assigns → `AmbiguousAssign`; assign to an undeclared root → `UnknownRoot`. Resolution is a single new assign whose `supersedes` invalidates the losers — no new event kind.
- **Chosen: cross-root references are protected from GC via pins.** Cross-root `kino://` refs are normal composition; an explicit `pin` (or an implicit pin from another root's composition reference) keeps a referenced version alive regardless of the owning root's policy.
- **Rejected: kino-declared membership / silent conflict resolution.** Both undermine the exclusive-ownership invariant and the fail-loud posture.
- **Not a dead end:** the `supersedes` resolution pattern is designed to generalize to future metadata-conflict resolution.
