# Spec kino content model: markdown prose + fenced code

A `kudo::api-spec` kino must carry both the human behavioral contract and the
machine signature. This spec settles how that content is structured and split.

## The decision
- **Chosen: a markdown blob = prose contract followed by fenced code, over a
  structured/typed schema.** Prose before/around fenced blocks is the doc
  contract; ```rust fenced blocks are the signature code (concatenated, in
  order). Stencil splits it via `pulldown-cmark` (already a kinora dep): prose →
  `///` doc-comments, fenced code → the signature body. It renders cleanly in
  mdbook too. Language-agnosticism lives in the pluggable target, not the kino
  format.
- **Chosen: offset-based excision** — `into_offset_iter` locates rust code
  blocks; their byte ranges are excised from the prose, their inner text
  collected as fragments. Non-rust fences and indented code stay in the prose.
  Blank lines are collapsed only at excision seams, so non-rust block interiors
  survive byte-for-byte.
- **Rejected: a typed/structured spec schema (separate doc + signature
  fields).** A single markdown blob is the natural authoring unit, renders in
  mdbook, and keeps the format language-agnostic. The split is a parse concern,
  not a storage concern.
- **Not a dead end:** the model already handles backtick-bearing docs — a
  definition containing ` ```rust ` uses a 4-backtick fence; prose containing
  inline ` ```rust ` renders literally (validated during the dogfood,
  [[kinora-3guj]]).

## What this built
- The `spec` module: `SpecItem { doc_prose, code_fragments }`, `parse(&str)`
  (infallible), `from_bytes(&[u8])` (UTF-8 decode, `SpecError::NotUtf8`),
  `code()` / `has_code()`. Implementation: [[kinora-q28s:task]].
