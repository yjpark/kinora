# `kinora repack` composes commit + clone-to-temp + atomic swap

A single command that rewrites the content store into a cleaner form, so users
don't drive three commands and manage a temp directory by hand. Named to echo
`git repack` and reserved as the future home for pack-file generation.

## The decision
- **Chosen: repack = commit → clone-to-temp-sibling → two-rename atomic swap.** Clone into `.kinora.repack-tmp` (same parent so the swap is one-filesystem renames), then rename `.kinora`→`.kinora.repack-old`, rename tmp→`.kinora`, unlink old. The two renames are the critical section; on rename-2 failure, roll back rename-1.
- **Chosen: unlink the old `.kinora` immediately after swap** — content is losslessly preserved in the new dir and git history is the real safety net; no intermediate backup dir.
- **Chosen: no no-op fast path** — always run commit + clone + swap; if the result is byte-identical, git shows no diff. Simpler than a dirty-check.
- **Chosen: refuse to run if a lingering `.kinora.repack-tmp` or `.kinora.repack-old` exists** — something went wrong last time and needs manual attention, not silent overwrite.
- **Chosen: if any root failed to commit, refuse to swap** — repack does not adopt `commit_all`'s per-root non-short-circuit behavior; swapping a partially-stale state is hard to reason about, and the user can re-run after fixing.
- **Not a dead end:** when pack files land, repack is the natural place to trigger pack generation (the clone walk already enumerates all reachable kinos) — a store-layer concern, not a new command.

## What this built
- `kinora::repack::repack_repo` (preflight guards, commit_all, clone-to-temp, two-rename swap + rollback) and a `kinora repack` CLI respecting the global `-C`. Verified by 6 library + 7 CLI tests (empty-repo success, lingering-sibling preflight errors, pending-staged-committed-before-swap, no-lingering-siblings-after-success, legacy filename rewrite, summary rendering, error cases).
