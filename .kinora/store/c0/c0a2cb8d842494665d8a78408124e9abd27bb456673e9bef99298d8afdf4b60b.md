# Ledger format: per-lineage append-only JSONL, content-addressed filenames

Settles RFC-0003 Q2 for the kinora bootstrap: the on-disk ledger format and file layout.

## The decision
- **Chosen: JSONL on-disk (one JSON event per line, append-only); facet for internal serialization.** JSONL is line-oriented (fits append-only), universal (recoverable without special tooling), and handles kinograph list-valued entries naturally.
- **Chosen: a directory of per-lineage append-only files under `.kinora/ledger/`, no privileged trunk.** Per-lineage files make git merges structurally conflict-free — every merge is an additive file change.
- **Chosen: lineage filename `<shorthash>.jsonl`** where `shorthash` = first 8 hex of BLAKE3 of that lineage's first event; `.kinora/HEAD` tracks the current lineage. Content-addressed filenames match Kinora's ethos and avoid branch-name coupling.
- **Rejected: tab-separated / single trunk file.** A single trunk reintroduces merge conflicts; TSV is poorer for list-valued data.
- **Not a dead end:** the bootstrap layout is explicitly transitional — the canonical hot-ledger + chained root-snapshot model supersedes it (see [[kinora-w7w0:spec]] for the post-bootstrap direction).

## What this built
- First `store` on a new git branch mints a new lineage; subsequent stores append. Realized by the `ledger` module (`mint_and_append`, `append_to_head`, HEAD tracking) in [[kinora-5k13:task]].

## Consequences / follow-ups
- Append-only invariant is enforced at the OS level (mint uses `create_new(true)`; append never creates).
- Branch-aware minting (`git rev-parse` integration) deferred; bootstrap reduces to "HEAD absent ⇒ mint, else append".
