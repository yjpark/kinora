# Task: Reword store `lineage=` print for the hot-ledger layout

**Goal.** On every successful `kinora store` the CLI printed `lineage=<sh> (new
lineage)` — carryover from the old per-lineage ledger. Under the hot-ledger layout
there is no lineage file and "lineage" misleads users (it reads as a branch-like
concept; it is really the event-hash prefix).

**Done when:** the store success line uses `event=<sh>` wording, `(new event)`
appears only on a genuinely new event (idempotent re-stores print no suffix), and a
test pins the new wording.

## Outcome
- CLI print reworded to `stored kind=<k> id=<id> hash=<h> event=<sh> (new event)`;
  `(new event)` suffix only when `was_new_lineage=true`.
- Formatter extracted as `format_store_summary(&StoredKino) -> String` in
  `crates/kinora-cli/src/store.rs` for unit testing without a real store.
- Three tests pin wording: new-event wording, idempotent-restore suffix omission,
  expected shape.
- `StoredKino.lineage`/`was_new_lineage` field names kept for one release so
  programmatic callers aren't broken; deprecation documented inline in
  `crates/kinora/src/kino.rs`. No docs/README changes needed.
