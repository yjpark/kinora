# Inline-only kinos via `inline=true` metadata

Composing a post as a kinograph needs post-specific connective prose
(intro/outro) that belongs to that post only — it must be a kino to sit in the
kinograph, but it should never render as a standalone page. This settles how that
signal is expressed.

## The decision
- **Chosen: a metadata flag `inline=true` over a dedicated kind.** Reusing the
  reserved-key metadata mechanism keeps the kind vocabulary small and lets any
  markdown kino opt out of page emission without a new type. Only the exact string
  `"true"` suppresses the page; `inline=false` and other values render normally.
- **Rejected: a dedicated `inline` kind.** A new kind would fork the markdown
  rendering path and force every connective-prose author to pick a non-obvious
  kind; the flag is additive and discoverable.
- **Not a dead end:** if inline semantics grow richer (e.g. per-context inlining
  rules), the flag can be promoted to a structured value or a kind later.

## What this built
- `crates/kinora/src/namespace.rs`: registered `inline` as a reserved bare
  metadata key.
- `crates/kinora/src/render.rs`: `render()` skips page emission for any kino whose
  head metadata has `inline=true`. Inlining is unaffected — `Kinograph::render`
  resolves members by id independently of page emission, so inline-only members
  still inline into composing kinographs.
- Verified: inline kino not emitted as a page; inline-only member still inlines
  while a normal sibling keeps its page; `inline=false` still renders a page.
