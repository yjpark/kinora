# Root GC policy as a long/short-term waterline

A proposed model for root garbage collection: a waterline where specs stay durable
(above water, never collected) and tasks sink (short-term, collected after done).
Observed as a clean fit during dogfooding; deferred until the broader policy design
settles. Not yet committed.

## The decision
- **Proposed: model root GC as a waterline** — specs = `never` (durable, above
  water); tasks = `maxage`/`keep-last-N` (short-term, sink after done. So a root's
  retention policy expresses how long-lived its members are meant to be.
- **Rejected (for now): redesigning the policy in isolation.** It should be designed
  alongside the task-lifecycle work (kino://617c9ea4901281677ce1fb21a4c3da413d427f0ed9acddd4365f751e5c10cc85) under the architecture
  epic kino://7d1a10e95dcee077fe2cc80f492c9f6ff299b490342800ab65f80f811a35e894, because "done tasks sink" is the same question as task
  lifecycle's "how do done tasks sink".
- **Not a dead end:** the supporting machinery largely exists already
  (`RootPolicy::Never`/`MaxAge`/`KeepLastN`, head_ts-based entry GC) — the open work
  is the user-facing model and config surface, not the mechanism, so adopting the
  waterline later is low-risk.
