# Kinograph composition format: styx `entries[]` of id-authoritative references

Kinographs are the composition kind: a `kind: kinograph` kino whose content is a styx `entries[]` list, each entry referencing another kino by its stable identity. This spec fixes that format and its store/render semantics.

## The decision
- **Chosen: styx content with `entries[]`, each entry `{id, name?, pin?, note?}`.** `id` is the authoritative kino-id reference; `name` is a non-authoritative hint preserved on round-trip; `pin` freezes the reference to a specific content hash; `note` is short commentary. facet-styx's inline form is used: `entries ({id b3…, name …, pin …, note "…"})`.
- **Chosen: store-time name→id rewrite.** Authors may write names or ids; `kinora store kinograph` resolves names to ids against current ledger state and saves content with ids filled in (authoritative) while preserving name hints for raw readability. Renaming a referenced kino later does not break the kinograph.
- **Chosen: pin == content hash.** Because `event.hash = BLAKE3(content)` in the store path, the event-hash matched by `resolve_at_version` IS the content hash — no separate content-hash field needed.
- **Chosen: render walks `entries[]` in order, resolving each `id` (or pinned hash), inlining content, with per-entry `note` rendered as a leading blockquote.**
- **Rejected: a YAML block shape and `Option<String>` entry fields.** facet-styx supports only its inline form and can't round-trip `Option<String>` (serializes `None` as `@`, rejects `@` on parse) — so optional fields are `String` with an empty-means-absent convention.
- **Not a dead end:** recursive rendering of nested kinographs is a stretch goal (warn for now); a current-name-drift lint can later compare preserved name hints against the current name without re-authoring.

## What this built
- The kinograph kind alongside markdown/text/binary, and the store-time rewrite. Realized in [[kinora-zboo:task]]; consumed by the renderer in [[kinora-9nom:task]]. Name resolution reuses the resolve semantics from [[kinora-6zxd:task]].

## Consequences / follow-ups
- Metadata on a kinograph event: `title`, `description`, optional `entry_notes` keyed by kino id, plus namespaced extensions.
