# Hybrid test scaffolding: doctests / named scaffolds / prose-only

A proposed RFC evolution (not yet committed) for stencil to render tests from a
spec kino's contract, choosing the form by contract shape.

## The decision (proposed)
- **Proposed: a hybrid test-rendering strategy** —
  - simple contracts → cargo-runnable doctests;
  - complex contracts → read-only-signature `#[test] fn name() { todo!() }`
    scaffolds whose bodies the agent fills (the signature is stencil-owned, the
    body editable — same split as functions in kino://4fe343db0e11af818156ab10c05f32841a1aa10078d60ac8973ecdabd5a2dada);
  - everything else → a prose-only fallback.
- **Rejected: a single uniform test form.** Doctests can't express complex setup;
  always-scaffolds adds noise for trivial contracts. A hybrid matches the
  contract to the cheapest faithful test form.
- **Open / not a dead end:** the *derivation rules* — which contract pattern maps
  to which test form — are themselves an open question, deferred. Filed as a
  forward direction.

## What this would build
- A test-target rendering path that classifies each spec kino's contract and
  emits the matching test form into a (read-only / partly-editable) region.
