# Canonical history is a chain of root kinograph versions, with `root` as a reserved kind

Settles kinora's self-hosting model: history is stored using the kinograph primitive itself, rather than a separate cold-ledger format. This is the core insight of the xi21 architecture.

## The decision
- **Chosen: canonical snapshots are chained `root`-kinded kinograph versions, over a separate compacted "cold ledger" format.** A root kinograph's entries enumerate its kinos (directly, or via sub-kinographs). Each version's `parents[]` points to previous version(s) of the same root's lineage. The structure parallels git: root kinograph ≈ commit, kino blob ≈ blob, sub-kinograph ≈ tree. Kinora self-hosts its own history using its own primitive.
- **Chosen: `root` is a reserved kind** joining `markdown`, `text`, `binary`, `kinograph` as a core (non-namespaced) kind. `ext_for_kind("root") -> Some("styx")` — roots are kinographs. Reserving it enables type-level invariant enforcement and specialized tooling.
- **Chosen: root entries use a richer flat leaf form** (`id`, `version`, `kind`, inline `metadata`, optional `note`, optional `pin`) distinct from the slim composition-entry form (`id`, `version?`, `note?`). The version field pins a specific content hash, giving the root true snapshot semantics; the inline `kind` lets render skip a second lookup.
- **Rejected: a separate cold-ledger compacted JSONL format.** Self-hosted root kinographs reuse the existing primitive and tooling instead of inventing a parallel format.
- **Not a dead end:** phase 4 introduces a second entry shape (`entry { subkinograph <kg-id>, version <hash> }`) for Merkle grouping inside roots, enabling O(log N) diffs on large repos. Only the flat form shipped initially.

## What this built
- A dedicated `RootKinograph` module (`root.rs`) with `RootEntry`/`RootKinograph`, parse/serialize via facet-styx, canonical sort-by-id, duplicate-id rejection, and hash/kind/metadata-key validation. `root` reserved in the namespace with the styx extension. See kino://18ed3234fb37fe7d6d4a4a85db155acdfba59e2ce278fc910e164c4a60ac8217.
- Implementation departed from the original "extend `Entry` with a variant / auto-detect by presence of `kind`" sketch: root and composition entries are distinct rust types because their `pin` semantics differ (boolean GC-exemption vs version hash) and callers always know which kind they are parsing.

## Consequences / follow-ups
- The root blob was later made self-contained by inlining commit metadata as a header line; see kino://7c184eb700cbdf99b47ca33d6a9d55fb0ae1af3fc0470e265b2cf45c72d30006.
- Per-entry `head_ts` was added later so GC could read retention timestamps off the entry instead of the event stream; see kino://fcb88cc171aecfffae7fa5785d35769d11e5d807291e79e982e4b87c024d754f.
- Multiple named roots, ownership, and per-root policy build on this; see kino://50d1260aba15f848bb3656667a42f08ebcff7e3bcf3fb1d815f24ae6399ddd24 and kino://344d509b8e3e8a39cd1bf7f2846f383e976bf5821764a2223503aa0e9431e32e.
