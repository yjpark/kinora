# Stencil marker protocol, region model, and LanguageTarget trait

Stencil needs a way to anchor its read-only blocks in source without parsing the
host language. This spec settles the line-based marker protocol, the file region
model, and the language-target abstraction.

## The decision
- **Chosen: a line-based marker protocol over language-aware parsing (silp
  ethos).** The comment leader comes from the language target (`//` for Rust):
  - `// stencil:kinograph <name|id>` — file-level binding, once per file.
  - `// stencil:slot <entry-name>` — the durable, agent-placed anchor.
  - `// stencil:ro <entry-name> <hash>` — stencil-written; opens the read-only
    block. `<hash>` is the resolved content hash, enabling no-op detection and a
    "read-only region was hand-edited" warning.
  - `// stencil:end` — closes the read-only block.
  On sync, stencil (re)writes the `ro…end` block immediately following each
  `slot`; it creates one if absent, refreshes it if present.
- **Chosen: model a file as an ordered sequence of blocks** — editable Text,
  Binding, Slot, ReadOnly. `parse → to_source` round-trips stencil-written input
  byte-for-byte (editable Text preserved verbatim; markers normalized — stencil
  owns them).
- **Chosen: a `LanguageTarget` trait (comment leader, doc-comment formatting,
  read-only block emission) with a shipped `RustTarget`.** New languages are
  additive with no engine redesign — satisfies RFC principle 3.
- **Rejected: AST/language-aware region detection.** Heavier, per-language, and
  contrary to the silp line-based ethos; the line protocol is enough to own
  read-only regions losslessly.
- **Not a dead end:** the marker spelling is refinable; the hash in the `ro`
  marker is a cheap precursor to full hash-checked enforcement
  ([[kinora-artb]]).

## What this built
- `target.rs` (`LanguageTarget` + `RustTarget`) and `region.rs` (`Block`,
  `StencilFile`, line parser, byte-stable `to_source`), wired into
  `StencilError`. Implementation and verification: [[kinora-thow:task]].
