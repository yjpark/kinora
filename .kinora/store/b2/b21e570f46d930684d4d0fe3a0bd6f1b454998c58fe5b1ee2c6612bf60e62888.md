# Kinograph kinos use the styxl one-entry-per-line format

Kinograph and root-kinograph blobs were serialized by `facet_styx::to_string`
into a single multi-thousand-char line — unreadable and undiffable. This settles
the on-disk format and the migration path.

## The decision
- **Chosen: styxl — a JSONL-style layout where each line is an independently-parseable inline-form styx document for one entry.** The file IS the sequence; there is no outer `entries ( ... )` wrapper. Applies to kinograph kinos (kind `kinograph`) and root kinographs; written with the new `.styxl` extension.
- **Rejected: the original `FormatOptions::default().multiline()` writer switch.** Empirically proven insufficient for our shape: facet-styx 3.0.2 forces `inline_start: true` for non-root structs/seqs, so `field_key`/`before_value` short-circuit (`struct_is_inline = inline_start && !force_multiline`) and `ForceStyle::Multiline` is bypassed. Newlines appear only between the root struct's direct fields; nested content stays inline. The `format_source` post-pass also won't help — `cst_format.rs` decides layout from CST separator tokens, not `force_style`.
- **Why styxl wins:** no upstream dependency (per-entry inline serialization works today), stream/grep/tail friendly (`head -1` is one valid entry), minimum-delta diffs (no wrapper context lines), and a cheaper implementation (two small `to_styxl`/`parse_styxl` wrappers over existing `to_string`/`from_str` on `Entry`).
- **Chosen: config.styx stays unchanged** — it is not a list-of-entries shape; single-line output is not a pain point there.
- **Not a dead end:** the alternative paths (upstream fix to `ForceStyle::Multiline`, fork/vendor facet-styx) remain available if styxl ever proves limiting. A future top-level header (schema version) can be reserved as the first styxl line, distinguished from entries by the absence of an `id` field.

## What this built
- `Kinograph::to_styxl`/`parse_styxl` and `RootKinograph::to_styxl`/`parse_styxl`, kind→extension routing to `.styxl` (the store's `find_blob_path` strips extensions, so content-addressed lookups survive the transition), and the `kinora reformat` migration that walks reachable kinograph kinos and stages `.styx`→`.styxl` versions. Verified by round-trip/edge-case tests, per-line parse-independence, and reformat-then-commit-makes-it-head tests.

## Consequences / follow-ups
- Depends on the `--repo-root` flag [[kinora-jezf]].
- The reformat migration is the realizing work — see [[kinora-tx3e:task]]. Root rewrites route through `store_kino(kind: "root", ...)` (not a direct blob+pointer bypass) so the event-graph linkage stays intact.
- Reformat's interaction with drained staging is handled by [[kinora-wcpp:task]] (synthesize stubs from root entries) and [[kinora-ou1u:task]] (nested composition entries via pins).
