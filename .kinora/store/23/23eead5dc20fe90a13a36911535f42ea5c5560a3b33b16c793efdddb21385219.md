# Public-API drift linter + hash-checked read-only enforcement

A proposed RFC evolution (not yet committed) upgrading stencil's read-only
regions from convention markers to enforced, drift-aware contracts.

## The decision (proposed)
- **Proposed: detect drift between rendered read-only sections and the actual
  exported items, and upgrade from convention markers to hash-checked (then
  possibly compile-failing) read-only regions.** Also warn on unmarked
  deprecations. The `<hash>` already carried in the `stencil:ro` marker
  (kino://4fe343db0e11af818156ab10c05f32841a1aa10078d60ac8973ecdabd5a2dada) is the cheap precursor — this turns the existing
  hand-edit warning into real enforcement.
- **Rejected (status quo): convention-only markers with a soft warning.** A
  warning is ignorable; a drift linter / compile-failing region makes the kino
  model the enforced source of truth for the public API.
- **Open / not a dead end:** how far to push enforcement (lint warning →
  CI-failing → compile-failing) is a graduated choice, not settled here; filed
  as a forward direction, low priority.

## What this would build
- A `stencil lint`-style check comparing exported items to the rendered
  read-only sections, surfacing drift, hand-edits, and unmarked deprecations.
