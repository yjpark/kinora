# Store blobs carry a kind-derived extension; reads are extension-agnostic

Extensionless store blobs are annoying to open directly while dogfooding. This
settles how on-disk filenames get a human-friendly extension without changing
dedup or read semantics.

## The decision
- **Chosen: derive the extension from kind at write time** — `markdown`→`.md`, `text`→`.txt`, `kinograph`→`.styx`, `binary`→none, namespaced `prefix::x`→`.bin`. Files are written to `<hash[..2]>/<hash>.<ext>`.
- **Chosen: readers resolve by 64-hex hash via a stem scan (`find_blob_path`), treating the extension as decoration.** Rejected recomputing the extension from the ledger event's kind on every read (strict but an extra lookup) — the extension is a UX hint, not a key, so glob/stem-match matches that framing.
- **Chosen: dedup semantics unchanged — same content under different kinds still produces one blob (first writer wins the extension).** The ledger event records kind for correctness; the on-disk extension is advisory.
- **Chosen: no migration of legacy extensionless blobs.** `find_blob_path` resolves both layouts, so there's no read-path pressure; re-storing would duplicate ledger events under new timestamps.
- **Not a dead end:** a rare multi-writer race on same-content/different-kinds could leave two blobs differing only in extension — semantically harmless (hash is the unique id, all readers stem-match one). Worth fencing with a lock only if multi-agent concurrent writes become real.

## What this built
- `ext_for_kind` in `namespace.rs` (reserved-kind → extension map) and `find_blob_path` in `paths.rs` (shard-dir scan matched by hash stem); `ContentStore::write(kind, bytes)` now picks the extension while `read`/`exists` use `find_blob_path`. The tmp filename was changed from `<hash>.tmp` to `.tmp-<hash>[.ext]` so a crashed partial write can't be mistaken for a real blob by the stem scan. Verified by per-kind round-trip, glob read, same-content-different-kind dedup, namespaced-kind fallback, and the tmp-name pin test.

## Consequences / follow-ups
- Parent epic: [[kinora-w7w0]].
- The extension-agnostic `find_blob_path` is what lets [[kinora-b1mg:spec]] (clone) tolerate legacy extensionless blobs and rewrite them to the current convention, and lets [[kinora-tx3e:spec]] introduce the `.styxl` extension without breaking content-addressed lookups.
- RFC-0003 re-import to produce `.md` on disk was explicitly deferred — legacy blobs stay readable; no automatic migration.
