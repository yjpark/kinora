# Extract content-addressing into standalone crates

Ahead of a major refactor, this settles how kinora's content-addressing core
(BLAKE3 identity + sharded blob store) is factored out. It is self-contained, used
pervasively (21 files), and has no intrinsic dependency on kinora's domain concepts
(kinds, ledger, kinographs). Design-only contract; the refactor itself is tracked by
[[kinora-kn71:task]].

## The decision
- **Chosen: two generic crates — `content-hash` and `content-store` — over keeping
  it in-tree or naming them `kinora-hash`/`kinora-store`.** Generic names allow reuse
  and independent publishing/testing, and give the planned pack-file work a clean
  home.
- **Chosen: decouple the store from `kind`.** `ContentStore::write(content, ext:
  Option<&str>)` replaces `write(kind, content)`; the `ext_for_kind` mapping stays in
  kinora (`namespace.rs`) and call sites pass the advisory extension. Dedup is by
  hash regardless of ext (first-writer-wins), so a pure CAS no longer couples to
  kinora's kind vocabulary.
- **Chosen: keep `Hash` as a `String` repr now.** Zero call-site churn —
  `as_hex() -> &str` stays. The `[u8;32]` repr optimization is a separate follow-up.
- **Chosen: design the pack-file backend seam now, ship loose-only.** A
  crate-internal `BlobBackend` trait (`LooseFileBackend` today) lets future pack files
  be interchangeable without any public-API change.
- **Chosen: a kinora re-export shim during migration** (`pub use content_hash::Hash;`)
  so existing `kinora::hash::Hash` paths keep working — avoids mass import churn.
- **Rejected: a hard cut-over without the shim,** and **deferring the pack seam
  entirely** — both would force avoidable churn or a later breaking API change.
- **Not a dead end:** the `Hash` repr and the pack backend can each evolve behind the
  preserved public API in separate follow-up epics.

## What this built
- A migration contract, staged so each step keeps the workspace green:
  1. Create `content-hash`; move `Hash`/`HashParseError`/`SHORTHASH_LEN`; kinora
     re-exports from its `hash` module.
  2. Create `content-store` (dep `content-hash`); move `ContentStore` + store-layout
     path helpers (internal `layout` module); change `write(kind)` ->
     `write(content, ext)`; update kino.rs (compute ext) + clone.rs
     (`copy_to`/`iter_hashes`); kinora re-exports `store`.
  3. Introduce the `BlobBackend` seam internally (`LooseFileBackend`); no public
     change.
- Public `Hash` API kept identical (`of_content`, `as_hex`, `shorthash`, `shard`,
  `FromStr`/`Display`) so the 21 call sites don't churn.

## Consequences / follow-ups
- Acceptance for the refactor: both crates compile standalone with their own tests;
  kinora depends on them; full workspace green; zero behavioral change
  (hash-preserving); clean per-stage commits.
- The `PackBackend` + repack-to-pack work is a later, separate epic that builds on
  the seam designed here (relates to the planned pack-file format).
- What stays in kinora: the `kind` vocabulary + `ext_for_kind`, repo-layout paths
  (staged/roots/config/kinora_root), and the re-export shims.
