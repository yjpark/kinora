# Task: Unassigned revisions inherit their owning root

**Goal.** A metadata-only rename (re-store same content with `--id`, no `--parents`,
no assign) bumped the ledger head but routed to the default `inbox` instead of the
root the kino already lived in, so the committed root kept the stale name and the
kino split across roots.

**Realizes:**
- kino://07bdd420655039ff9170e069be132ddfc68a548ab8f0d6c93a6f23f6c64b8191 — Preserve lineage through the root kinograph.

**Done when:** a re-store of an already-committed kino with no fresh assign inherits
the root that currently owns it; a brand-new unassigned kino still defaults to
`inbox`; the committed root reflects the live name.

## Outcome
- `route_kino` (commit.rs): an unassigned kino inherits the root that currently owns
  it (via a kino-id -> root map from prior committed roots), falling back to `inbox`
  only for brand-new kinos. Explicit live assigns still win.
- `collect_root_ownership` builds the ownership snapshot once per batch, threaded
  through `commit_all`/`commit_root`/`commit_root_with_refs`/`build_root`.
- Archive (`collect_owned_staged_events`) and prune (`prune_staged_events`) now use
  the same `route_kino` routing, fixing a prior inconsistency that pruned events a
  root never committed.
- Semantics: only revisions of already-owned kinos inherit; brand-new unassigned
  kinos still default to `inbox` — unifies with the kino://07bdd420655039ff9170e069be132ddfc68a548ab8f0d6c93a6f23f6c64b8191 fork fix so a
  bare revision no longer splits across roots.
- Tests: metadata rename updates the committed root (not inbox); unassigned revision
  inherits prior root while brand-new defaults to inbox. Full workspace green, zero
  warnings.
