# Hash algorithm: BLAKE3 plain hash, 64-char hex, 2-char sharding

Settles RFC-0003 Q1 for the kinora bootstrap: how content is hashed and addressed on disk.

## The decision
- **Chosen: BLAKE3, plain (unkeyed) hash, 32-byte output as 64-char lowercase hex.** Stable public spec, first-class Rust support, WASM/no-std friendly, optimized for tree hashing, fast.
- **Rejected: SHA-256 for `sha256sum` ubiquity.** Kinora ships its own CLI, so external-tool ubiquity is less critical than the above properties.
- **Not a dead end:** the hash is encapsulated behind a `Hash` newtype; swapping the algorithm is a localized change if ever needed.

## What this built
- Content-store paths shard by the first 2 hex chars: `.kinora/store/aa/aabb…`.
- Realized by the `hash` module (`Hash` newtype with `shorthash()`, `shard()`, strict `FromStr`). See [[kinora-5k13:task]].

## Consequences / follow-ups
- Lineage filenames and shorthashes reuse the same BLAKE3 digest, first 8 hex chars.
