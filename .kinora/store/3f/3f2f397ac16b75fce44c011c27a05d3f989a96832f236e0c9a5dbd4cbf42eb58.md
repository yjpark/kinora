# Concurrency & conflict handling: structural file merges + detect-at-resolve forks

Settles RFC-0003 Q4 for the kinora bootstrap: how concurrent ledger appends and version forks are handled.

## The decision
- **Chosen: ledger file-level merges are structurally conflict-free** (per-lineage files from kino://c0a2cb8d842494665d8a78408124e9abd27bb456673e9bef99298d8afdf4b60b), so concurrent git branches never collide at the file level.
- **Chosen: version-DAG forks are detected at resolve time** — branch-aware (HEAD's descendant wins) with refuse-and-report fallback; bootstrap is detect-and-report only.
- **Chosen: reconciliation is a regular version event with multiple `parents[]`** — no new event kind. Manual reconcile in MVP.
- **Chosen: metadata merge is per-field, ts-latest-wins; events carry only changed fields; `null` removes a field. Timestamps are RFC3339 UTC** with no clock-skew handling (documented assumption).
- **Rejected: CRDT-style per-element merging, logical clocks, interactive reconcile UX.** Out of MVP scope — too much machinery for the bootstrap; the multi-parent reconciliation event covers the real need.
- **Not a dead end:** CRDT set-union (e.g. tag merge) and logical clocks can be layered in later without changing the event envelope.

## What this built
- Write-time validation (namespace rules + parent/ref existence + kino-id consistency) and resolve-time fork detection.
