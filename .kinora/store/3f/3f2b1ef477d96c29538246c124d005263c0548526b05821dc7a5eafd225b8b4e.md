# Commit archives staged events into a per-root `commit-archive` kino

When a root's version bumps during commit, the staged events it consumed are
serialized into a durable archive kino so the provenance trail survives staged
cleanup. This settles how that archive is identified, formatted, and where it
lives.

## The decision
- **Chosen: one archive kino per root per version-bump, identity = the root's new_version hash.** Each archive's parent is the previous archive for that same root — mirrors per-branch commit tracking in git. Rejected a per-run hash because there is no canonical per-invocation hash; the root version hash is already canonical.
- **Chosen: reserved `commits` root, auto-provisioned, hardcoded by name.** Mirrors the existing `inbox` pattern — `Config::from_styx` re-injects `commits` if absent, user can override its policy but not remove it. Rejected a fully user-declared root (more discoverable but adds migration churn).
- **Chosen: reuse `RootPolicy::Never` for the `commits` root** rather than inventing a `keep-all` variant — "never drop anything" is exactly archive semantics.
- **Chosen: archive wire format is JSONL with a schema header line** (`{"@schema":"kinora-commit-archive-v1"}` then one `Event.to_json_line()` per line). Cheap future-proofing; body stays greppable.
- **Chosen: the `commits` root special-cases its own version bumps** — it skips archive-of-the-archive creation (the commits kinograph already records each archive) and drops the transient archive-assign events without archiving them. Rejected committing `commits` recursively.
- **Chosen: render excludes the `commits` root via hardcoded name check** (`if root_name == "commits" { skip }`), consistent with other infra name lookups. Rejected a general infra-root config marker as YAGNI.
- **Not a dead end:** per-run queries ("what did this invocation do?") are synthesizable later via a `kinora log` command (timestamp-cluster or stamp a run-id); a general infra-root marker can replace the hardcoded checks if ever justified.

## What this built
- The `kinora::commit_archive` module (v1 JSONL serialize/parse with a hand-rolled header reader) and `maybe_archive_owned_events` in the commit pipeline: after a non-commits root promotes work, it collects owned staged events, serializes, stores via ContentStore with kind `commit-archive`, and writes an assign into `commits`. `commit_all` iterates `commits` last with a fresh ledger re-read. Verified by end-to-end tests (archive + assign produced, commits root doesn't archive itself, commits kinograph contains the archive, repeated runs idempotent via content-addressing + live-assign guards).

## Consequences / follow-ups
- Staged cleanup itself was deferred so the archive flow could land and be exercised first — see [[kinora-bayr:spec]] (prune-after-archive + prior-root merge for Never), then generalized to MaxAge in [[kinora-wcpp:task]].
- Depends on the hot→staged / compact→commit rename bean [[kinora-2t6l]].
