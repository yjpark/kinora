# Typed non-composition links between kinos + reverse lookup

The only structural link between kinos today is the kinograph — an ordered
composition meant for rendering/inlining. There is no primitive for a plain typed
reference (task->plan, task->spec, code->spec), and no reverse lookup ("what
references this kino?"). This proposes a typed, non-composition link primitive with
reverse traversal. Not yet committed.

## The decision
- **Proposed: a typed `links` metadata value** as a compact typed list, e.g.
  `spec:<id>,plan:<id>` (type token + target id), with each target validated as a
  64-hex id (or a name resolved at store time, like kinograph entries). The `links`
  bare metadata key is already reserved in `namespace.rs` but has no defined
  semantics yet.
- **Chosen: links are NON-composition.** Unlike kinograph pins they must NOT
  pin/protect their targets from GC — the resolver/commit GC ignores them. This is the
  key distinction from the existing composition primitive and must be confirmed.
- **Rejected: encoding these as prose/`kino://<id>` content references only.** That
  is the status quo: untyped, with no reverse index. Resolved via
  `render::rewrite_kino_urls` for page links but not queryable.
- **Proposed reverse lookup:** `Resolver::referrers(id) -> Vec<(referrer_id, type)>`
  scanning every identity head's `links` metadata (O(n) over identities is fine at
  current scale; an index can come later). CLI surface: `kinora links <id>`
  (outgoing) and `kinora backlinks <id>` (incoming), honoring `--json`
  ([[kinora-n6eg:task]]).
- **Not a dead end:** a read-only reverse index over existing `kino://<id>` content
  references could ship independently first, delivering "what references this kino?"
  with zero schema commitment if the typed-link design stalls.

- Nothing yet — a captured design proposal; status `draft`.
